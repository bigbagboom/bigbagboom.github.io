记录一次折腾openwrt全过程

Date 二 09 九月 2025 Tags Linux / OpenWRT / QEMU / KVM / R76S / Apple Airport Time Capsule

距离上次记录折腾电脑的经历已经过去半年，期间岁月静好，出门旅游了几次，AI继续爆发，但我却毫无钻研的兴致。然而友善发布的一款新软路由设备改变了这一切。

大概是在7月初的时候，开始在网上看到友善电子r76s的信息，我从21年初开始用友善的r2s，而r2s实际在2020年初就发布了，距今已超过5年，对一款cpu定死的小主机来说，5年显然已经落后时代很多（但实话实说，r2s直到现在也一直在卖，价格也没比我买它时便宜多少），因此我对r76s的出现很兴奋，就开始格外关注。

从硬件规格来说，r76s比r2s要强上很多，不去抄网上的参数，就用我看到的benchmark来比，r76s所用的cpu跑分差不多是r2s芯片的4倍，再加上r2s的内存只有1G，而r76s则有的选，分别是2G和4G，网络也有升级，r2s是双千兆网卡，而r76s则是双2.5Gbps。当然，硬件上去了，价格也就上去了，r76s目前售价差不多是r2s的一倍，考虑到r2s依然在卖，这个价格也算合理，当然和多年前cpu换代的水平显然不太一样，在我心里，相差5年的硬件，性能升4倍，价格应该和5年前的持平才对，摩尔定律应该还管用不是吗？但这恐怕就是我的一厢情愿，摩尔定律也许早就不管用了，英特尔也早已不是昔日的芯片巨人。

r76s出来了，但我并没有第一时间剁手，其中一个重要的原因是系统。因为r76s相对较新，所以openwrt官网和immortalwrt官网都没有第一时间提供支持的固件，友善自己确实提供了一个openwrt的定制版friendlywrt，但我入手r2s时用friendlywrt不太满意（后来换了第三方编译的openwrt），从此有了偏见，于是这次就想要再等等。但过了快一个月，网上关于固件的信息毫无变化，出于冒险的心态，我最终还是在8月初下单了4G内存版的r76s。

没有别的选择，所以只能用友善自己的固件，但没想到的是，友善的版本用起来也还好。至少我希望有的功能它都提供，最关键的wireguard/科学上网/docker都没什么问题，我一度感觉直接将它替换掉r2s完成家庭网络升级已经指日可待。直到，出现了一个完全意想不到的状况。（这里需要插一段，使用友善friendluwrt一段时间后，我手贱更新了它内部软件库中提供的一些组件，正如某位网友提醒的那样，千万不要随意在openwrt中尝试更新，果然这次不知道哪个组件的更新搞坏了系统web gui luci，管理界面起不来等于整个系统废掉，本来安装openwrt而不是更纯粹的Linux系统就为了各种为路由而定制的管理界面方便设置，如果崩掉只能全部重来。这时我在网上看到已经有第三方编译的openwrt开始支持r76s，虽然openwrt/immortalwrt官方依然不支持，我还是下载来试用了一番，同样因为忍不住升级又搞坏了几次系统，重刷系统固件的过程令人崩溃，但最终我还是选定这个第三方编译固件而不是友善自己的，可能还是出于对友善的信心不足吧。这里需要再次强调，OpenWrt软件包不要升级，不要升级，不要升级！）

在使用r2s作为家中科学上网主力设备的同时，r2s其实也承担着家庭娱乐中枢的功能，这主要是靠alist和jellyfin这两大影音神器实现的。在r76s上安装alist和jellyfin自然不是问题，问题在于我家里用于存储影音文件的网络存储设备是一台用了十多年的苹果Airport Time Capsule! 在2025年的今天，很多人也许不知道这是个什么东西，但在当年，它算是苹果相当受欢迎的一款产品，结合了千兆无线路由以及高达3T的家庭文件存储能力，既能作为苹果电脑的时间机器备份系统，又能作为一台轻NAS提供网络共享文件功能。而我，秉承着充分利用现有设备的理念，在使用软路由系统搭建影音服务器的时候，将它作为网络文件存储的后端发挥着余热。而这几年，它一直工作良好。

但r76s改变了这一切。确切地说，是新版的openwrt系统改变了这一切。更准确地说，是内核5.15及以上版本的Linux系统改变了这一切。新版Linux内核禁止了Time Capsule中不安全的网络共享连接协议，导致原先可用的mount命令在新系统中一直提示错误，原本轻松可达的网络共享开始变得异常困难。（神奇的是，哪怕是最新版的win11，Time Capsule的连接也始终不是问题，在最新Mac系统中Time Capsule的连接也毫无障碍，据说到2026年，新版Mac系统将不再支持使用Time Capsule备份，但那只是因为AFP协议不再被支持，SMB网络共享估计还是可以的。）

事实上，不光是r76s，就算是r2s，如果安装了2022年后新出的openwrt版本，也是无法连上Time Capsule的网络共享文件夹的。因为我的r2s系统Linux内核一直停在5.4，这几年除了折腾上面的各种软件，系统本身并没去折腾，所以这个问题始终未显现。在了解到新内核有这样的安全限制后，我试着用我的LinuxMint桌面系统去访问Time Capsule，果然也一样连不上，但就像我在前面说过，最新版的Win/Mac均没有这种困扰，看起来Linux这种前瞻性安全限制行为对我们这种拥有老旧设备又无法获得软件升级（苹果停止了该产品线的生产，也不再提供新的软件支持）的人实在是太不友好了。

话说回来，如果r76s没法使用影音服务，那这次硬件升级效果必将大打折扣，等于是把每天都要用到的关键服务砍掉，这显然是我无法接受的。于是，我只能先让r2s继续服役，然后想其他办法解决这个问题。

首先，解决方案肯定是有的，最显而易见的，是升级老旧的Time Capsule到更现代化的NAS系统，这个不是没有想过，但如果一开始就考虑这个方案，r76s也许就不该买，因为现在有很多NAS方案其实是可以把软路由功能集成进去的。说到底，很多NAS系统底层就是一个Linux系统，只要硬件足够强，什么功能都可以有（这个方案未被实行的另一个原因是经济上的，买NAS不算冲动消费，但毕竟价格不菲，在当前经济形式下，省钱方案始终是王道。）。第二个方案就是继续使用现有资源，想办法绕过新版Linux内核的安全限制，这是一个相当有挑战性的想法，也是我折腾这么多天，终于有素材可以写成本文的原因。

为了实施第二方案，我的第一步行动就是打开Google，打开AI，虚心求教：如何绕开Linux升级到新版内核无法访问Time Capsule的限制？网上搜索没啥成果，估计和我一样有这种奇葩要求硬件条件又相似的人并不太多。确实有人在抱怨，但背景条件往往不太一样，给出解决方案的人也几乎没有，也没看到任何反馈说成功了的，总之就是搜索不到。AI那里则自信地多，看AI的回复能一下子给出5、6个方案，但真的照着做下来才发现都不靠谱。尤其是有一次AI给出了一个看似非常合理的方案：使用docker安装旧版的linux系统，利用旧版系统连接Time Capsule，然后再通过共享目录暴露给宿主机。听起来是不是很Promising? 在尝试了多次依然失败后我突然意识到，Linux下docker是和宿主机共享系统内核的（也许Win/Mac下不是，但我这OpenWrt肯定是），所以不管我装多旧的系统（我从ubuntu20.04开始装起，试了18.04,16.04甚至14.04后才醒悟），内核始终是新的，所以也就始终受新内核的安全限制约束。另一个方案是网上一位程序员开发的程序叫afpfs-ng，就是为了绕开这个安全限制访问Time Capsule的AFP文件系统而写，这个程序能下载到的二进制代码是基于ubuntu x86的，为了在arm架构的r76s下能用，我费了很大力气在r76s的OpenWrt中搭出了编译环境，从源码开始编译，克服了各种编译时发生的错误，最终终于成功编译出可执行代码，但在使用时却发现完全连不上也没反馈任何信息，毕竟是十多年前的代码，能编译成功已算难能可贵，达不到期望的功能毫不令人奇怪，也可能是我用法不对，但我折腾的时间够长了，实在没力气继续在它上面纠缠，就此放弃。

最终，我能尝试的只剩AI给出的一个最复杂的方案：在r76s上使用qemu虚拟机安装旧版Linux。这个方案当初在一众AI给出的回答里是最不受待见的，因为当时其他方案看上去更简单也更有望成功，但到了最后，全村最后的希望还是落到了qemu身上，事实证明，最后的希望往往能力挽狂澜。同时也说明，AI还是有点用的，但在各种不靠谱中，这份有用很难一开始就被识别出来。

虽然最终证明qemu的方案是可行的，但实施过程中还是经历了各种波折，所以值得详细记录一番。

首先说明一下，qemu是linux环境下开源的虚拟机系统，它的优点是可以纯软件模拟不同的硬件架构，比如它在x86下可以模拟arm，反过来当然也行，这一点是其他类似软件比如vmware或virtualbox都不具备的能力。但同时，如果搭建和宿主机同样架构的虚拟机，则和其他虚拟机软件一样，qemu可以利用底层硬件虚拟化的能力，极大提升虚拟机的运行效率。同时，由于qemu是模拟一个完整的操作系统，就不会发生像docker那样共享宿主内核的情况，系统完全从底层开始搭建，原来系统用什么内核安装出来就是什么内核。这也是这个方案能成功的关键：只有使用5.15之前的内核系统才能绕过新版Linux内核的安全限制，从而正常访问我老旧的Time Capsule共享硬盘。

openwrt最新版的软件仓库里有qemu-system-x86_64和qemu-system-aarch64，分别是用来开x86/64虚拟机和arm虚拟机。不知道出于什么考虑，我首先尝试的是安装x86虚拟机，也就是用qemu-system-x86_64安装debian10系统，之所以用debian而不是更熟悉的ubuntu是因为ubuntu的云主机镜像的root密码我没搞定，看了网上的文档也找不到怎么获得或更改密码的地方，而debian的云主机镜像则非常友好，root直接就是没有密码的。我相信没有用云服务器的人会傻到设置完系统不更新密码，所以ubuntu搞那么麻烦在我看来是毫无必要。但我既然学艺不精，也就只能哪个简单选哪个，好在，对于我所需要的功能，选debian和ubuntu其实并没有什么差别。另外，debian10的内核是5.10，我试过下载一个debian11的镜像，发现内核是6.1，也许更早的debian11版本有较低版本的内核，但后来也懒得找了。

创建x86虚拟机的代码是我从网上抄来的，运行起来没什么问题，就和其他虚拟机一样，开机，系统引导，加载各种初始化程序，最后进入登陆提示界面。

qemu-system-x86_64 -smp 1 -m 1G -drive file=debian-10-backports-nocloud-amd64-20220526-1027.qcow2,if=virtio \
-netdev user,id=eth0,net=192.168.1.0/24,hostfwd=tcp::10022-:22,hostfwd=tcp::5244-:5244 \
-device virtio-net-pci,netdev=eth0 \
-nographic

使用root不用密码进入debian，设不设密码无所谓，反正自己家里用，在NAT后面相对也算安全。网络不需要任何设置直接能ping通宿主机所在的局域网，也能依靠NAT连接到互联网，尝试mount命令挂载Time Capsule的共享文件夹，成功！当然，之间还是有一些挫折，比如需要安装一些原本没有的程序发现apt用不了，上网查了原来是debian10太老，很多更新源已经不维护移除了，解决方案是改成debian11的几个源，我不知道为什么可以这样，反正最后是成功了。

最大的问题解决，接下来当然是测试应用。直接在虚拟机中安装Alist，前面的虚拟机启动命令已经将Alist的端口映射到宿主机上，这样从家里的局域网就能直接打开浏览器访问到，一切都没什么问题，直到发现.......播放的视频是会卡的。原先使用r2s，哪怕播放4K高码率视频都毫无压力，而现在只是打开普通的1080p视频就会卡，这显然不能让人接受。于是，接下来就需要找到问题并解决它。

r76s的硬件指标高出r2s一大截，即使使用虚拟机有一定损耗也不应该出现如此夸张的性能问题，所以我觉得肯定是哪里做错了。调整方案的第一步，就是把qemu x86虚拟机换成arm的。这一调整的逻辑很明显，我在一台arm架构的小主机上，用x86虚拟机显然需要进行指令集的翻译，而arm虚拟机应该就没这层开销，那不就提高程序运行效率了么？这么想当然是没错，但接下来的问题，一个比一个费神。首先，把qemu-system-x86_64换成qemu-system-aarch64，后面的参数不变直接运行，就会报错，查网络，创建arm虚拟机，需要加-M和-cpu参数，网上有不少例子，照着依样画葫芦加完，命令果然不再报错，但系统也一样起不来。我一度以为是下载的arm版云主机镜像有什么问题，换了几个别的比如debian11的也一样启动没反应，百思不得其解后，再次上网去查，发现别人的arm虚拟机启动命令会多加载一个叫QEMU_EFI.fd镜像，研究下来是用于arm系统启动引导的，看来arm系统和x86不太一样，x86可以不需要单独系统引导就能启动OS，而arm则不行。于是照着网上的指点下载到了专门用于引导的镜像QEMU_EFI.fd，把加载参数加到命令中，这下系统便启动起来了。之后和x86的一样，登陆，改apt源，装软件，挂载共享文件夹，装Alist，测试，然后，然后，然后发现播放视频依然还是卡。从x86换到arm，性能完全没有改善，或者说，完全感觉不到有什么变化。

问题还会在哪呢？一开始，我的排查重心在虚拟机的整体性能上，是不是分配给虚拟机的cpu核心和内存不够？我给启动命令增加了核心数和内存数，但结果是一样的，说明提高虚拟机的硬件资源和目前遇到的问题无关，那么，什么才是相关的？在苦思冥想挺长一段时间后，我忽然意识到，播放卡顿，会不会和cpu性能无关，而只是单纯的网络问题？如果网络带宽不够，视频就需要不断暂停加载缓冲，自然就会卡了。猜对了方向，要查就简单了。进入虚拟机系统，安装iperf3，连接局域网中的r2s做传输速度测试，结果真的让人大跌眼镜。r76s和r2s直接用iperf3测，大概在940Mbits/sec的水平，考虑到家里较差的网络布线水平，这个数值基本符合千兆网络设备的应有表现。但从虚拟机内部再测，这个数值一下子变成了23Mbits/sec，竟然只有宿主机网络性能的 1/40，这还只是以虚拟机作为客户端连接r2s服务端测出的数值，反过来（虚拟机作为服务端）连我当时没测，但后面会提到，反过来测得到的数据会更低，可能只有一半左右。

定位到真正的问题，接下来就是着手解决的过程。但这个过程依然不那么顺利。

作为qemu毫无疑问的新手，我对于提高虚拟机的网络性能是完全没有概念的。遇事不决问AI，这一次，AI给出的答案挺靠谱。AI提到，在我的启动命令中，虚拟机使用user模式的网络性能较差，改成桥接可以大幅改善。但改桥接的话，原先命令中的端口映射不再可用，端口映射需要改成在宿主机上用iptables命令去模拟，那样的话，虚拟机就不能再使用DHCP方式获取IP地址，因为万一增加虚拟机，它的IP就不再是确定的，IP地址必须采用静态地址在虚拟机中指定，以便宿主机上iptables命令使用，修改debian10的IP地址对我来说也是新知识，好在不算特别难。这里记录一下在debian10内部改静态地址的代码：

root@debian:/etc/network# cat cloud-interfaces-template
auto $INTERFACE
#allow-hotplug $INTERFACE

#iface $INTERFACE inet dhcp
iface $INTERFACE inet static
 address 192.168.2.128
 netmask 255.255.255.0
 gateway 192.168.2.1
 dns-nameservers 223.5.5.5

这些都改完之后再次测试，速度果然上去很多，提高到差不多230Mbits/sec，播放1080p是完全不卡了，但放高码率4K视频依然会卡，其实光从230Mbits/sec这个数值来看似乎播放4K不应该再卡，但这次我反过来测了一下，因为Alist事实上是从虚拟机里提供服务，所以虚拟机是服务端，将虚拟机设为iperf3服务端后从r2s用iperf3连接虚拟机映射出的端口，这次数值从230Mbits/sec降到了110Mbits/sec左右，也就是相当于百兆网络的水平。考虑到我的播放过程首先是让虚拟机从网络共享文件夹读取视频文件，然后再通过Alist对外提供视频流，网络带宽可用的还需要再减半才行，再加上些其他运行损耗，播放高码率4K视频继续卡就不奇怪了。

搞半天还是没最终解决，十倍的网络性能提升依然不够用，说实话是让人有点气馁的。继续问AI，这次AI提到使用qemu虚拟机要提高性能必须确保openwrt启用kvm。然而，使用AI给出的查看系统是否已启用kvm的命令 lsmod|grep kvm，r76s的输出却是一片空白，感觉像是系统并不支持kvm，我记得r76s当初的商品介绍中是提到能够支持kvm的，我下载的第三方编译的openwrt也说系统编译了支持kvm的模块，但我始终找不到任何开启或证明它支持kvm的地方。僵持了一段时间之后，我无意中在网上看到一条答复qemu问题的帖子，回答者结合-cpu和-accel参数解决了原提问者的问题，我之前使用的-cpu参数是指定了某款具体的arm芯片型号，但没有加-accel参数，网上的那个回答提到只需要将cpu指定为host并将accel设定为kvm就行，于是，我把完整的qemu启动命令改成下面这样：

qemu-system-aarch64 -M virt -cpu host -accel kvm -smp 1 -m 1G -nographic \
-bios QEMU_EFI.fd \
-drive file=debian-10-backports-nocloud-arm64-20240703-1797.qcow2,if=virtio \
-netdev bridge,br=br-lan,id=br0 \
-device virtio-net-pci,mq=on,vectors=2,netdev=br0

是的，我不光改写了-cpu的参数，增加-accel，还把内核数和内存都改小了，事实证明，只要网络性能提升，虚拟机并不需要太高的硬件资源就完全能胜任我要求它的工作。经过这条启动命令执行后的虚拟机，iperf3测试速度直接达到了733Mbits/sec，这还是以虚拟机作为服务端的速度。反过来，则基本能跑满宿主机的带宽，网络性能瓶颈再也不存在了。再次用视频服务测试，但这次换成了openlist，据说是Alist卖身之后失去了用户的信任，不管如何，播放高码率的4K视频是再也不卡了。大大大成功。
